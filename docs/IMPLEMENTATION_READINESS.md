# Implementation Readiness

## 오늘 정리할 항목

1. 요구사항을 구현 가능한 단위로 쪼갠다.
2. InsightChat 안에서 이 기능이 어느 레이어에 들어가는지 가설을 세운다.
3. 기존 선행 구현에서 재사용할 수 있는 모듈 범위를 정리한다.
4. 내일 구현 시작 전 필요한 접근 권한과 환경 의존성을 점검한다.

## 요구사항 기준 핵심 구현 축

- 비동기 작업 접수와 상태 조회
- URL별 콘텐츠 수집
- OCR 포함 콘텐츠 보강
- LLM 기반 키워드 생성
- 결과 파일 저장과 다운로드
- 완료 알림
- URL 단위 격리와 부분 완료
- 동일 URL 캐시

## 내일 구현 시작 전 체크리스트

### 권한/접근

- [ ] InsightChat 메인 프로젝트 저장소 접근 권한
- [ ] AWS 계정/리전/실행 권한 범위
- [ ] Bedrock 모델 사용 가능 여부
- [ ] 결과 저장용 버킷 또는 기존 저장소 확인
- [ ] 알림 발송 수단(email/webhook) 구현 위치 확인

### 코드베이스 확인

- [ ] InsightChat의 API 진입점 위치
- [ ] 비동기 작업 프레임워크 존재 여부
- [ ] 기존 queue, worker, scheduler 사용 패턴
- [ ] 공통 로깅/모니터링/에러 처리 방식
- [ ] 기존 DTO, response schema, auth 미들웨어 규칙

### 기능 설계 확인

- [ ] 요청/응답 스키마 확정
- [ ] 작업 상태 모델 확정
- [ ] URL별 결과 JSON 스키마 확정
- [ ] CSV 내보내기 포맷 확정
- [ ] 캐시 키와 TTL 저장 방식 확정

### 콘텐츠 수집 재사용성 확인

- [ ] Crawl4AI 기반 수집 코드를 그대로 재사용 가능한지
- [ ] OCR worker 분리 배포가 필요한지
- [ ] 봇 차단/실패 패턴 JSON 자산이 이미 있는지
- [ ] 도메인별 JS 보정 캐시 로직이 있는지

## 메인 프로젝트 접근 후 먼저 확인할 질문

1. InsightChat은 이미 job orchestration 계층을 갖고 있는가?
2. 이 기능은 새 서비스인가, 기존 backend module 확장인가?
3. 결과 저장은 S3 같은 파일 저장소를 쓰는가, DB blob/link를 쓰는가?
4. 알림은 기존 notification service를 타는가?
5. 사용자 단위 접근제어와 작업 ownership 모델이 이미 있는가?

## 현재 기준 리스크

- 메인 프로젝트 구조 미확인 상태라 실제 폴더 구조와 런타임 제약은 아직 미정이다.
- `artifacts/` 문서는 2026-04-02 시점 기준이므로 실제 운영 코드와 차이가 있을 수 있다.
- Bedrock 모델명, 배포 리전, 기존 공통 인프라 방식은 접근 권한 확보 후 재검증이 필요하다.
