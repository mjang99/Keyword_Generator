# Keyword Quality Evaluation Criteria

This project now uses a deployed-first quality workflow.

- `scripts/evaluate_deployed_quality.py`
  - Primary acceptance gate
  - Evaluates artifacts generated by the deployed API
- `tests/evaluate_quality.py`
  - Reference-only evaluator
  - Used for local regression checks with fixtures

## Deployed-First Workflow

1. Submit a real job through the deployed API.
2. Wait until the job reaches a terminal state.
3. Evaluate the deployed artifact.

```powershell
python scripts/evaluate_deployed_quality.py --job-id job_0001 --api-base https://example.execute-api.ap-northeast-2.amazonaws.com/
```

If you already have an artifact, evaluate it directly:

```powershell
python scripts/evaluate_deployed_quality.py --combined-json artifacts\combined.json
python scripts/evaluate_deployed_quality.py --combined-json s3://bucket/jobs/job_0001/summary/combined.json --region ap-northeast-2
```

## Reference Workflow

Reference evaluation is still useful for fast local iteration, but it is not the final acceptance gate.

```powershell
python tests/evaluate_quality.py --fixture evidence_commerce_pdp_rich.json --platform both
python tests/evaluate_quality.py --fixture evidence_support_spec.json --platform both
python tests/evaluate_quality.py --fixture evidence_borderline.json --platform naver_sa
```

## Gate vs Scorecard

The evaluator now separates hard gating from diagnostic scoring.

### Hard Gate

The deployed job or per-URL artifact fails the gate when any of these conditions are true:

- Positive keyword count for the target platform is below `100`
- `filler_ratio >= 0.15`
- `avg_naturalness < 0.70`
- `semantic_unique_ratio < 0.95`

`quality_warning=true` does not fail the gate by itself. It is a caution flag, not a verdict.

### Scorecard

The scorecard is for analysis and A/B comparisons. It should not be treated as the only PASS/REWORK decision.

- `auto_score`
- `exact_unique_ratio`
- `semantic_unique_ratio`
- `reason_filled_ratio`
- `category_distribution_deviation`
- top duplicate families

## Metric Definitions

### filler_ratio

Detects obvious low-information patterns such as:

- keywords ending in numbered fallback variants
- repeated `검색`, `정리`, `카테고리`, `정보` style scaffolds
- legacy filler suffix families such as `특성 검색`, `브랜드 검색`, `문제 해결`

### avg_naturalness

Keyword naturalness is scored heuristically.

Penalties include:

- filler-like scaffolds
- repeated tokens
- extremely short keywords
- document boilerplate such as `지원`, `사양`, `spec`, `manual`
- excessive modifier stacking

Support/spec pages are allowed to keep valid long model names, so they are not penalized as aggressively as commerce pages.

### exact_unique_ratio

Exact string uniqueness. Useful for debugging but not enough on its own.

### semantic_unique_ratio

Near-duplicate uniqueness after normalizing out light modifiers such as:

- `추천`
- `후기`
- `리뷰`
- `비교`
- `정리`

This is the primary uniqueness gate because exact-string uniqueness misses shallow variations.

## Job-Level Interpretation

The deployed evaluator reports both:

- job-level verdict
- `successful_url_coverage`

This matters for `PARTIAL_COMPLETED` jobs:

- a partially completed job can still pass quality for its successful URLs
- low coverage is still surfaced so the runtime problem is visible

## Rollback Signals

Consider rollback or rework when one of these happens on the same smoke URL set:

- hard gate failures increase
- `semantic_unique_ratio` drops
- support/spec naturalness regresses
- successful URL coverage drops
- duplicate-family concentration increases

## Notes

- The old `auto_score >= 60` rule is no longer the primary acceptance gate.
- Score remains useful for trend comparisons, but hard gate metrics decide PASS/REWORK.
- Fixture evaluation remains valuable for fast regression detection before re-running deployed smoke tests.
