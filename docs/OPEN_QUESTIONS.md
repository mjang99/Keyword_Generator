- "both" 선택 시: 네이버 100개 이상 + 구글 100개 이상
- 단독 선택 시: 해당 플랫폼 100개 이상
- **구현 전략 확정** (2026-04-06): 공유 evidence pack에서 단일 LLM 호출 → 플랫폼별 렌더링. 설계 Section 14 참조.

### RQ-02: 부정 키워드(제외 추천)의 count 포함 여부 ✅ 확정 (수정)

- **답변**: **제외** (2026-04-06 재확인, 2026-04-03 최초 기록 오류 정정)
- 부정 키워드는 플랫폼별 100개 양성(positive) 키워드 count에 **포함되지 않는다**
- 부정 키워드는 별도 출력 항목으로 제공 (Naver: `제외키워드`, Google: `negative`)
- 설계 Section 4.2 이미 반영됨 (`negative` 카테고리는 positive floor 외부)

### RQ-03: 콘텐츠 충족도(sufficiency) 정량 기준 ⚠️ 잠정 적용

### RQ-04: URL 캐시 키 정규화 방식 ⚠️ 방향 확정, 세부 미정

- 상품 식별 파라미터(`?id=123`, `?product=abc` 등)는 유지
- 정확한 제거 파라미터 목록은 구현 시 결정

### RQ-08: `both` 모드 결과 row 구성 방식 ✅ 확정

- **답변**: **재량 처리, API 명세에 작성** (2026-04-06)
- 설계 Section 4.3에 `both` flatten 규칙 이미 확정:
  - 동일 키워드 텍스트 + 양 플랫폼 모두 허용 → 1 row (`naver_match` + `google_match` 모두 채움)
  - 한 플랫폼만 허용 → 1 row (비요청 플랫폼 필드 blank)
  - 플랫폼별 렌더링 키워드 텍스트가 다를 때 → 2 rows (각각 별도 row)

### RQ-09: 중복 판단 기준 및 후처리 방식 ✅ 확정

- **답변**: **LLM이 의미 중복 판단, 품질 평가, 보충까지 모두 처리. 퀄리티 1순위, 비용 최적화는 추후** (2026-04-06)
- `mens sneaker`, `men sneaker`, `men's sneaker` → **의미 중복(LLM 판단), 하나만 유지**
- 후처리 파이프라인: A(과생성) → B(LLM 의미 dedup + 품질평가) → C(LLM 보충, 필요 시만)
- 하드 규칙(프로모 위반, 가짜 가격, 재고 긴급성, 경쟁사 안전)만 deterministic 코드 유지
- 비용 절감은 퀄리티 베이스라인 검증 후 최적화 단계에서 탐색
- 설계 Section 4.2.4, 4.2.7 참조

### RQ-10: 키워드 언어 정책 ✅ 확정

- **답변**: **한국어 + 영문 혼합 허용** (2026-04-06)
- 브랜드명·모델명은 영문 유지 권장
- 일반 품목어·수식어는 한국어 사용 권장
- 같은 의미의 한국어·영문 표기 키워드는 **양쪽 모두 허용** (별도 키워드로 count)
  - 예: `나이키 슬리퍼`, `Nike 슬리퍼` → 각각 독립 키워드
- Keyword Generator 프롬프트에 혼합 언어 예시 포함 필요

### RQ-11: 비판매 페이지 실패 처리 기준 ✅ 확정

- **답변**: **LLM이 해당 URL이 제품 설명(판매) 페이지인지 판단하는 단계 추가** (2026-04-06)
- 차단/대기실/상품 없음: 기존대로 FAIL 처리
- 지원 페이지·기술사양 페이지처럼 판매 의도가 없는 URL: LLM 분류 단계에서 `non_product_page` 또는 `support_spec_page`로 명시 판단
- LLM 분류 프롬프트에 "이 URL이 실제 제품을 판매하거나 구매 유도하는 페이지인가?" 판단 항목 추가
- 현재 설계 Section 2.3.6 분류기 Bedrock tie-break 로직에 반영 필요

### RQ-12: 캐시 TTL 및 플랫폼별 재사용 정책 ✅ 확정

- **답변** (2026-04-06):
  - **TTL 30일**로 변경 (기존 7일에서 수정)
  - **Cache Validity Worker**: TTL과 별개로, 전용 스케줄 워커(daily)가 캐시 엔트리를 주기적으로 스캔하여 URL 유효성 확인
    - URL이 사라졌으면 (404/410) → 캐시 엔트리 삭제
    - 콘텐츠가 의미있게 변경됐으면 → 캐시 엔트리 삭제 (다음 요청 시 재생성)
    - Submit 시점에는 live URL 체크 없음 — 모든 사전 무효화는 이 워커가 담당
  - **플랫폼 캐시 partial reuse**:
    - `naver_sa` + `google_sa` 캐시 모두 존재 → `both` 요청 시 병합 후 `both` 캐시 저장 및 제공
    - `both` 캐시 존재 → `naver_sa` 또는 `google_sa` 단독 요청 시 해당 부분 추출해서 제공
    - 즉, 나중에 `naver_sa`만 재요청해도 `both` 캐시에서 꺼내줄 수 있음
  - 설계 Section 3.4 전면 수정 필요
