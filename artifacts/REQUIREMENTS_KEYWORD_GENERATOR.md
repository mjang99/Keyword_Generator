# 요구사항 명세서: URL 기반 검색 광고 키워드 자동 생성 시스템

**문서 버전**: 1.2  
**작성일**: 2026-04-02  
**프로젝트**: InsighChat — URL Product Keyword Generator

---

## 1. 프로젝트 개요

### 1.1 목적

상품/서비스 URL을 입력받아, 해당 페이지의 콘텐츠를 자동으로 수집·분석하여 **네이버 SA(검색광고) 및 구글 SA**에 즉시 사용 가능한 검색 광고 키워드를 대량으로 생성하는 완전 자동화 시스템을 구축한다.

### 1.2 배경

광고 캠페인 생성 시 키워드 발굴은 수작업으로 처리되어 시간이 과다 소요된다. 이 시스템은 URL만 제공하면 상품 특성, 경쟁사, 구매 의도 등을 반영한 키워드 100개 이상을 자동으로 생성하여, 광고 운영자가 캠페인 등록 시 즉시 활용할 수 있도록 한다.

### 1.3 시스템 처리 흐름

```
URL 목록 제출 (최대 30개)
    │
    ▼
콘텐츠 수집 (웹 스크래핑 + 이미지 OCR)
    │
    ▼  수집 실패 URL 격리
    │
키워드 생성 (LLM)
    │
    ▼
결과 저장 + 사용자 알림
```

| FR-04 | 요청 시 완료 알림을 받을 이메일 주소 또는 webhook URL을 선택적으로 지정할 수 있다. |

### 2.2 콘텐츠 수집

| ID | 요구사항 |
|----|---------|
| FR-05 | URL별로 상품 상세 페이지의 텍스트 콘텐츠를 수집한다. JavaScript 렌더링 기반의 동적 페이지를 지원해야 한다. |
| FR-06 | 상품 이미지에 포함된 텍스트(소재, 성분, 기능 설명 등)도 OCR을 통해 추출하여 콘텐츠에 포함한다. |
| FR-07 | 수집된 콘텐츠는 상품명, 가격, 옵션 정보와 상품 설명 본문을 포함하는 형태로 정제한다. |
| FR-08 | 콘텐츠 수집에 실패한 URL(봇 차단, 상품 삭제, 렌더링 실패 등)은 키워드 생성 단계로 넘어가지 않고 실패 목록에 기록된다. |
| FR-09 | 수집된 콘텐츠가 키워드 생성에 충분한지 자동으로 판정한다. 불충분한 경우 추가 수집을 시도한다. 추가 수집 후에도 불충분하면 품질 경고와 함께 키워드 생성을 진행한다. |

### 2.3 키워드 생성

| ID | 요구사항 |
|----|---------|
| FR-10 | URL당 **최소 100개** 이상의 키워드를 생성해야 한다. |
| FR-11 | 아래 §2.4에 정의된 10개 카테고리를 모두 포함하여 생성한다. |
| FR-12 | 네이버 SA용 키워드와 구글 SA용 키워드를 각각 구분하여 제공한다. |
| FR-13 | 각 키워드에는 **카테고리**, **매칭 유형**(네이버: 정확/확장, 구글: exact/phrase/broad), **생성 근거**를 포함한다. |
| FR-14 | LLM은 **AWS Bedrock Claude Sonnet 3.5**를 기본으로 사용한다. |
| FR-15 | 동일 URL에 대해 재요청 시 이전 결과가 있으면 캐시에서 반환하고 LLM을 재호출하지 않는다 (TTL: 7일). |

### 2.4 키워드 카테고리 정의

| 카테고리 | 설명 | 예시 (등산화 상품) |
|---------|------|----------------|
| **브랜드 키워드** | 브랜드명 + 제품 라인 | 노스페이스 등산화, 노스페이스 트레킹화 |
| **일반/카테고리 키워드** | 제품 유형, 상위 카테고리 | 등산화, 트레킹화, 아웃도어 신발 |
| **세부/속성 키워드** | 소재, 기능, 사이즈 등 구체적 특성 | 방수 등산화, 고어텍스 트레킹화, 경량 등산화 |
| **경쟁사 키워드** | 주요 경쟁 브랜드명 포함 | 살로몬 등산화 비교, 콜롬비아 트레킹화 |
| **구매 의도 키워드** | 구매/비교/추천 의도 포함 | 등산화 추천, 등산화 구매, 남성 등산화 순위 |
| **롱테일 키워드** | 3어절 이상 구체적 검색어 | 발볼넓은 남성 방수 등산화, 초경량 트레일 러닝화 추천 |
| **혜택/가격 키워드** | 할인, 가격대, 쿠폰 관련 | 등산화 할인, 10만원대 등산화, 등산화 최저가 |
| **시즌/이벤트 키워드** | 계절, 행사, 상황 기반 | 봄 등산화, 겨울 방한 등산화, 단풍 시즌 트레킹 |
| **문제/솔루션 키워드** | 고객 고충 기반 검색어 | 발목 보호 등산화, 미끄럼방지 등산화, 무릎 충격 완화 |
| **부정 키워드 (제외 추천)** | 광고 낭비 방지를 위한 제외 권장어 | 등산화 AS, 등산화 수선, 중고 등산화 |

## 4. 오류 처리 및 부분 완료

### 4.1 URL별 실패 유형과 처리

| 실패 유형 | 발생 단계 | 처리 |
|---------|---------|------|
| 봇 차단 | 콘텐츠 수집 | FAIL 기록, 키워드 생성 스킵 |
| 상품 없음/삭제 | 콘텐츠 수집 | FAIL 기록, 키워드 생성 스킵 |
| 렌더링 실패 | 콘텐츠 수집 | FAIL 기록, 키워드 생성 스킵 |
| 콘텐츠 불충분 (추가 수집 후에도) | 콘텐츠 수집 | 부분 결과로 키워드 생성, 품질 경고 표시 |
| LLM 키워드 생성 실패 | 키워드 생성 | 재시도 후 FAILED 기록 |
| 키워드 수 부족 (< 100개) | 키워드 생성 | 추가 생성 1회 재시도 |

| 컬럼 | 설명 |

| product_name | 상품명 |
| category | 키워드 카테고리 |
| keyword | 키워드 |
| naver_match | 네이버 매칭 유형 (정확/확장/제외) |

## 7. 제약사항

1. **봇 차단 쇼핑몰**: 서버단에서 headless 브라우저를 차단하는 쇼핑몰은 콘텐츠 수집이 불가하며 FAIL로 처리된다.
2. **상품 상세 페이지 전용**: 홈페이지, 카테고리 목록 페이지 등은 지원 대상이 아니다. 개별 상품의 상세 페이지 URL만 입력한다.
3. **키워드 품질 보증**: 콘텐츠 불충분 상태에서 생성된 키워드는 품질이 저하될 수 있으며, 품질 경고 플래그로 사용자에게 안내한다.

# 부록 A. 콘텐츠 수집 모듈 선행 실험 결과

> Stage 2 (콘텐츠 수집) 구현을 위한 참고 자료.  
> 총 6개 버전의 프로토타입을 거쳐 아래 결론을 도출하였다.

## A.1 스크래핑 엔진 비교 실험

| 방식 | JS 렌더링 | 콘텐츠 품질 | 비용 | 결론 |
|------|---------|-----------|------|------|
| 유료 스크래핑 API (Firecrawl) | 지원 | 양호 | 유료 (API 호출당 과금) | 외부 의존성, 비용 부담 |
| Playwright 직접 제어 + markdownify | 지원 | 보통 | 무료 | 노이즈 필터링 별도 구현 필요 |
| Trafilatura | 미지원 | 보통 | 무료 | JS 미지원으로 SPA 대응 불가 |
| **Crawl4AI (Playwright 기반)** | **지원** | **우수** | **무료 (Apache 2.0)** | **채택** |

**채택 근거**: Crawl4AI는 JS 렌더링을 지원하고, PruningContentFilter로 노이즈가 제거된 LLM 소비용 마크다운을 자동 생성한다. 무료이며 Lambda 컨테이너 배포가 가능하다.

## A.2 OCR 필요성 검증

상품 상세 페이지의 핵심 정보(소재, 성분, 기능 설명 등)가 이미지로만 제공되는 경우가 많다. 텍스트만 추출 시 키워드 생성에 필요한 정보가 누락된다.

- **채택 OCR**: PaddleOCR (한국어 지원, CPU 실행, 오픈소스)
- **필터링 기준**: OCR 텍스트 30자 이상인 이미지만 포함 (간판/로고 등 노이즈 제외)
- **도메인별 이미지 패턴 학습**: LLM이 "유용한 이미지"의 URL 패턴을 학습하여 캐싱. 재방문 시 불필요한 이미지의 OCR을 생략.

## A.3 콘텐츠 정제 방식 (Hybrid 출력)

PruningContentFilter(노이즈 제거) 적용 시 가격/옵션 정보가 함께 제거되는 문제가 발견되었다.

**해결**: 두 영역을 결합한 Hybrid 방식 채택.
- **헤더 영역** (raw 마크다운): 상품명 ~ 가격 클러스터 끝 + 500자 버퍼, 최대 5,000자
- **본문 영역** (PruningContentFilter 적용): 노이즈가 제거된 상품 설명
- 이 두 영역을 결합한 텍스트를 LLM 입력으로 사용

## A.4 실패 감지 및 자동 대응

| 실패 유형 | 감지 방법 | 자동 대응 |
|---------|---------|---------|
| 봇 차단 | 키워드 패턴 매칭 (캡챠 문구 등) | 스텔스 모드 재시도 1회 |
| 상품 삭제/품절 | 키워드 패턴 매칭 | FAIL 즉시 처리 |
| 렌더링 불완전 | 추출 텍스트 < 3,000자 | 팝업 제거 + 대기시간 연장 재시도 |
| 원인 불명 | LLM 진단 (Bedrock) | 원인 분류 후 처리 |

새로운 실패 키워드 발견 시 JSON 파일로 저장 → 이후 LLM 호출 없이 즉시 감지 (자가 학습).

## A.5 콘텐츠 충족도 검증 및 자동 보정

- **검증**: 수집 결과를 Bedrock에 "검색 광고 키워드 생성에 충분한가?" 판정 요청
- **자동 보정**: 불충분 판정 시 스크린샷 → Bedrock vision으로 누락 정보 파악 → 사이트별 맞춤 JS 코드 생성 → 재수집
- **JS 캐싱**: 생성된 JS 코드는 도메인별 캐싱. 재방문 시 즉시 적용하여 LLM 호출 절약.

# 부록 B. 선행 구현 AWS 인프라 설계 (참고)

> 콘텐츠 수집 모듈을 AWS Lambda에 배포한 기존 설계. 키워드 생성 파이프라인 설계 시 참고.

### B.1 기존 Lambda 구성

| Lambda | 역할 | 배포 방식 |
|--------|------|---------|
| Dispatcher | URL 목록 수신 → SQS 메시지 분배 | 일반 Lambda |
| Scraper Worker | URL별 웹 스크래핑 + 실패 감지 | 컨테이너 이미지 (Playwright 포함) |
| OCR Worker | 이미지 다운로드 + PaddleOCR 실행 | 컨테이너 이미지 (PaddleOCR 포함) |

### B.2 단계 간 연결

```
API Gateway → Lambda Dispatcher → SQS → Lambda Scraper → SQS → Lambda OCR
```

- 각 단계에서 실패한 URL은 다음 단계로 전달되지 않음
- SQS Dead Letter Queue로 재시도 실패 메시지 격리
- DynamoDB로 URL별 처리 상태 추적

### B.3 컨테이너 이미지

Scraper Worker와 OCR Worker는 의존성 크기 문제로 Lambda 컨테이너 이미지(ECR)로 배포한다.
- Scraper: Playwright + Crawl4AI → 이미지 크기 ~1.5GB
- OCR: PaddleOCR + PaddlePaddle → 이미지 크기 ~2GB

### B.4 Bedrock 모델

| 용도 | 모델 | 리전 |
|------|------|------|
| 실패 진단, OCR 패턴 학습, 충족도 검증 | Claude Sonnet 3.5 | ap-northeast-2 |
| 최종 진단 에스컬레이션, JS 코드 생성 (vision) | Claude Sonnet 4.6 | ap-northeast-2 |
